# 做站 PRD 流程（通用版）

## 适用场景

用于把一个机会词 / 关键词 / 产品想法，转成可以交给文案、设计、开发执行的产品定义文档。

适合：
- SEO 工具站
- AI 工具站
- 小产品矩阵
- 独立开发者产品
- 从关键词出发的出海站点

不适合：
- 纯市场调研报告
- 只写 Landing Page 文案
- 没有明确关键词或用户需求的脑暴

---

## 输入要求

开始写 PRD 前，尽量准备这些信息：

| 输入 | 说明 | 没有怎么办 |
|---|---|---|
| 目标关键词 | 主关键词或机会词 | 先补关键词 |
| 月搜索量 / CPC / KD | 用于判断需求和商业价值 | 没有就标注为待验证 |
| SERP Top 10 | Google 前 10 结果 | 自己实搜补齐 |
| Top 3 竞品 | 直接竞品或相邻竞品 | 从 SERP 里选 |
| 用户痛点证据 | Reddit、评论、竞品评价、论坛 | 没有就不要编 |
| 趋势判断 | 需求是长期还是短期热点 | 用 Google Trends / Ahrefs / Semrush 辅助 |

原则：**证据不全可以继续，但必须标注假设；不能把猜测写成事实。**

---

## 工作流

### Step 1：值不值得做（5 分钟过滤）

用 4 个问题快速判断：

1. **长期需求还是短期热点？**  
   看 12 个月趋势，避免只追短期热点。

2. **SERP 里有独立小站能排进去吗？**  
   如果前 10 全是大站，难度高；如果有 3 个以上小站，有切口。

3. **用户有明确付费场景吗？**  
   竞品收费、用户抱怨贵、用户有高频刚需，说明可能有商业化空间。

4. **首版能不能低成本实现？**  
   优先做静态站、工具站、API 组合型产品；谨慎做需要训练模型、复杂队列、重运营的产品。

判定：
- 3-4 个强：继续做 PRD
- 2 个强：谨慎推进，先缩小范围
- 0-1 个强：暂缓或退回调研

---

### Step 2：SERP 实扫，验证搜索意图（15 分钟）

不扫 SERP 不写 PRD。

Google 搜目标词，看前 10：
- 每个结果是什么类型：工具页 / 信息页 / 对比文章 / 模板页 / 图片工具 / 文本工具
- 用户到底想完成什么任务
- 前 10 里大站和小站比例
- Top 3 竞品提供了哪些核心功能

| 搜索意图 | SERP 特征 | 产品方向 |
|---|---|---|
| 工具型 | 前 10 多数是在线工具 | 首屏必须可直接使用工具 |
| 信息型 | 前 10 多数是教程/解释/列表 | 先做内容页或工具+内容组合 |
| 对比型 | 大量 best / alternative / review | 做工具，同时布局对比页 |
| 模板型 | 大量模板、案例、示例 | 提供模板库或可复制结果 |
| 图片型 | 多数是图片生成/编辑工具 | 必须支持图片相关能力 |
| 文本型 | 多数是文本生成/处理工具 | 可以先做文本工作流 |

输出：
```markdown
### SERP 观察
目标词：xxx
搜索日期：YYYY-MM-DD
Top 10 类型分布：
- 工具页：x 个
- 信息页：x 个
- 大站：x 个
- 独立小站：x 个

搜索意图判断：xxx
产品方向：xxx
主要竞品：xxx / xxx / xxx
```

---

### Step 3：用户与场景

至少拆 3 个细分用户，再选 1 个主力用户。

| 维度 | 问题 |
|---|---|
| Who | 谁最痛？一句话画像 |
| Pain | 为什么现在就要解决？有多急？ |
| Current | 现在用什么替代方案？ |
| Trigger | 什么事件触发搜索？ |
| Hangout | 用户在哪里出现？Reddit / X / Discord / Google |
| Willingness | 愿意付多少钱？为什么？ |

三层竞品：

| 层级 | 定义 | 例子 |
|---|---|---|
| Tier 1 | 直接竞品，同品类 | SERP 上的同类工具站 |
| Tier 2 | 相邻方案，功能重叠 | ChatGPT、Canva、Notion、Midjourney 等 |
| Tier 3 | 现状 / 不做 | 手动处理、Excel、复制粘贴、不解决 |

重点：很多小产品不是抢 Tier 1，而是抢 Tier 3——把原来手动做的人变成用户。

---

### Step 4：产品定位

用 6 步定位法：

1. 列出竞争替代：Tier 1 / Tier 2 / Tier 3
2. 提取独有属性：功能、体验、定价、速度、场景
3. 属性 → 价值映射：每个属性解决什么问题
4. 锁定最佳用户：最痛、最容易触达、最愿付费
5. 选择市场品类：正面竞争 / 细分切入 / 创造新品类
6. 叠加趋势：为什么现在做

定位语句：
```text
FOR [目标用户]
WHO [具体痛点]
[产品名] IS A [产品品类]
THAT [核心价值]
UNLIKE [竞品/替代方案]
[产品名] [关键差异]
```

消息层级：

| 层级 | 内容 | 要求 |
|---|---|---|
| Headline | 5-10 个词核心承诺 | 3 秒内理解价值 |
| Subhead | 1 句话说清怎么做到 | 补充 how |
| Benefits | 3-4 个核心好处 | 结果导向 |
| Features | 支撑卖点的功能 | 功能 → 好处 |
| Proof | 证据 | 数据、案例、评价、样例 |

差异化优先级：
1. 更聚焦的单点定位
2. 更清楚的场景切入
3. 更低摩擦的免费使用
4. 更强的 SEO 页面结构
5. 更合适的定价和包装

不要只写“功能更多”。

---

### Step 5：功能规划

先判断做 MVP 还是完整版：

| 情况 | 策略 |
|---|---|
| 目标词有成熟竞品 + 你有团队产能 | 做完整版，上线即有竞争力 |
| 全新品类 / 搜索量很小 / 需求不确定 | MVP 先验证需求 |
| 技术风险高 / 成本不可控 | MVP 先验证核心链路 |
| 工具型 SERP 已成熟 | 不要用半成品上线 |

如果竞品已经成熟，半成品上线反而是劣势。用户来了发现功能残缺，会直接去竞品。

功能分类：

| 类别 | 包含 |
|---|---|
| 核心功能 | 用户完成核心任务的流程 |
| Landing Page & SEO | 首页、场景页、FAQ、博客/对比页 |
| 付费 & 转化 | 免费限制、升级提示、支付、Pro 解锁 |
| 合规 & 基础设施 | Privacy Policy、Terms、埋点、错误监控 |
| NOT-DO | 明确不做什么 |

NOT-DO 判断标准：
- 技术复杂度高但非核心
- 需要大量运营投入
- 和定位无关
- 竞品也没做，且用户不关心
- 会拖慢首版上线

---

### Step 6：页面信息架构（IA）

首页推荐结构：

```text
1. Hero
   - 一句话说清工具干什么
   - 首屏内嵌工具入口或核心 Demo
   - 主 CTA
2. Demo / 效果展示
3. How It Works（三步流程）
4. Use Cases（3-4 个使用场景）
5. Features（功能亮点）
6. Pricing（定价或免费说明）
7. FAQ（覆盖 Top 5-8 个顾虑）
8. Footer CTA
```

转化检查：
- [ ] Hero 是否 3 秒内说明价值
- [ ] 首屏是否能开始使用或看到 Demo
- [ ] CTA 是否是“动词 + 结果”
- [ ] 是否有 How It Works
- [ ] 是否有真实样例 / 结果展示
- [ ] FAQ 是否覆盖价格、质量、格式、隐私、限制
- [ ] 定价区是否先展示价值再展示价格

SEO 页面矩阵：

| 页面 | 目标词 | 目的 | 优先级 |
|---|---|---|---|
| / | 主词 | 转化 | P0 |
| /use-case-1 | 长尾词 1 | 场景页，SEO | P1 |
| /use-case-2 | 长尾词 2 | 场景页，SEO | P1 |
| /blog/best-xxx | 信息词 | 引流 | P2 |
| /alternative/xxx | 替代词 | 截流 | P2 |

### Step 7：域名调研（通用版）

目标：快速找到 3 个可注册、可读、和产品相关的域名候选。

#### 7.1 候选生成

每个关键词至少生成 10-15 个候选。

类型：

**直接关键词型**
- `{keyword}.com`
- `{keyword}.app`
- `{keyword}.tools`
- `{keyword}.dev`
- `{keyword}.co`

**前缀型**
- `get{keyword}.com`
- `try{keyword}.com`
- `use{keyword}.com`
- `my{keyword}.com`

**AI / Tool 组合型**
- `{keyword}ai.com`
- `{keyword}tool.com`
- `{shortword}ai.com`
- `{shortword}tools.com`

**品牌造词型**
- 两个短词组合
- 关键词缩写 + 动词
- 场景词 + 工具词

#### 7.2 查询方式

优先用域名查询工具：
- NameBeta
- QueryDomain
- Namecheap / GoDaddy / Porkbun
- Cloudflare Registrar

如果工具结果冲突，再用技术方式辅助验证：
- Whois
- DNS 查询
- HTTP 访问
- RDAP 查询

普通学员不需要一开始就跑四重验证。只有在准备购买或工具显示冲突时，再做交叉验证。

#### 7.3 后缀优先级

| 后缀 | 优先级 | 说明 |
|---|---|---|
| .com | 最高 | 最通用，最适合长期品牌 |
| .app / .tools / .dev | 高 | 适合工具站 / 开发者产品 |
| .ai | 中 | 适合 AI 产品，但价格更高 |
| .co | 中低 | 可做备选 |
| 其他后缀 | 低 | 除非非常贴合品类 |

#### 7.4 判定规则

| 状态 | 判定 |
|---|---|
| 工具显示可注册 + 价格正常 | 进入候选池 |
| 工具显示已注册 | 放弃 |
| 多个工具结果冲突 | 换工具复查或放弃 |
| 价格明显过高 | 放弃，除非是强品牌名 |
| 太长 / 难读 / 难拼 | 降级或放弃 |
| 涉及大品牌 / 商标 | 放弃 |

#### 7.5 推荐标准

优先选择：
1. 短
2. 顺口
3. 好拼
4. 和产品有关
5. 不容易和大品牌混淆
6. 价格可接受

不为了精确匹配关键词牺牲可读性。宁可选一个顺口的 `getxxx.com`，不要选一个又长又难拼的精确匹配域名。

输出格式：
```markdown
### 域名调研结果

目标关键词：xxx

| 域名 | 状态 | 价格 | 评价 |
|---|---|---|---|
| getxxx.com | 可注册 | $xx/年 | 推荐，顺口，适合品牌 |
| xxx.tools | 可注册 | $xx/年 | 关键词强，工具属性明确 |
| xxxai.com | 可注册 | $xx/年 | 可作为备选 |

推荐 Top 3：
1. xxx.com — 理由
2. getxxx.com — 理由
3. xxx.tools — 理由

最终建议：
优先注册 `xxx.com`。如果预算或可用性不合适，选 `getxxx.com`。
```

注意：域名状态有时效性。确定要做后，最好 24 小时内注册。

## PRD 交付物结构

最终 PRD 建议包含 11 个模块：

```markdown
# [产品名] PRD

## 0. 结论先行
- 要不要做
- 为什么做
- 一句话定位
- 首版做什么
- 明确不做什么

## 1. 市场概述
- 目标关键词
- 搜索量 / CPC / KD
- 趋势判断
- 机会判断

## 2. SERP 与竞品分析
- SERP Top 10 类型
- Top 3 竞品
- 三层竞品分级
- 用户痛点证据

## 3. 目标用户
- 3 个细分用户
- 主力用户
- 触发场景
- 付费意愿

## 4. 产品定位
- 定位语句
- 消息层级
- 差异化
- 禁词清单

## 5. 功能规划
- 核心功能
- SEO / Landing Page
- 付费与转化
- 合规与基础设施
- NOT-DO

## 6. 页面信息架构
- 首页结构
- 关键页面
- SEO 页面矩阵

## 7. 定价设计
- 竞品价格锚点
- 免费版限制
- Pro / Lifetime 方案
- 成本验算

## 8. 域名与技术栈
- 域名候选
- 推荐 Top 3
- 技术栈建议

## 9. GTM 策略
- 首发渠道
- 首周动作
- 内容计划

## 10. 转化漏斗与埋点
- page_view
- tool_start
- tool_success
- tool_error
- cta_click
- upgrade_click
- checkout_start
- payment_success

## 11. 风险评估
- SEO 风险
- 技术风险
- 成本风险
- 合规风险
- 商业化风险
```

## 交接摘要

### 给文案

```markdown
产品名：xxx
定位语句：xxx
Headline：xxx
Subhead：xxx
核心 Benefits：
1. xxx
2. xxx
3. xxx
FAQ 必须覆盖：xxx
不能把产品说成：xxx
CTA 格式：动词 + 结果
```

### 给设计

```markdown
首页结构：xxx
首屏重点：xxx
核心交互：xxx
视觉参考：xxx
移动端要求：xxx
不需要设计：xxx
```

### 给开发

```markdown
技术栈建议：xxx
P0 功能：xxx
P1 功能：xxx
API 链路：xxx
数据结构：xxx
NOT-DO：xxx
验收标准：xxx
```

## 质量检查清单

PRD 完成前自查：

- [ ] 有明确关键词
- [ ] 有搜索量 / CPC / KD，或明确标注待验证
- [ ] SERP 经过实扫
- [ ] 搜索意图判断清楚
- [ ] Top 3 竞品写清楚
- [ ] 竞品做了三层分级
- [ ] 至少拆了 3 个用户场景
- [ ] 选定主力用户
- [ ] 有结构化定位语句
- [ ] 消息层级清楚
- [ ] 功能范围分 P0 / P1 / NOT-DO
- [ ] 首页 IA 可以直接给设计使用
- [ ] SEO 页面矩阵明确
- [ ] 定价有竞品锚点和成本意识
- [ ] 域名有 Top 3 推荐
- [ ] GTM 有首周动作
- [ ] 埋点事件明确
- [ ] 风险有缓解措施

## 使用原则

- 先证明需求，再扩完整体系
- 成熟竞品市场，首版必须有完整价值闭环
- 不为了“看起来完整”堆功能
- NOT-DO 比 DO 更重要
- 所有事实要有来源；没有来源就标注为假设
- PRD 的目标不是好看，而是减少文案、设计、开发的反复沟通
